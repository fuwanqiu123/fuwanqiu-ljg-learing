* 问题

你采访了十个用户，记了五十页笔记，画了三张 empathy map。回到办公室，团队七嘴八舌——"用户说想要更快的按钮""不对，他们说配色难看""我觉得加个语音功能更好"。每个人都在讲自己的版本，没有两个人对准同一个靶子。

采访得越多，信息越杂。怎么把这些碎片拧成一根绳，让团队知道"我们到底要解决什么问题"？

* 翻译

** 锚点：酿酒

同理心阶段是去葡萄园里摘葡萄——越多越好，红的绿的酸的甜的都采回来。Define 阶段就是把葡萄倒进压榨机：去梗、压榨、发酵，最后得到一小桶能喝的酒。Module 3 教的就是这套"酿酒工艺"：怎么把满地的葡萄变成一瓶能卖的东西。

** 第一步：写问题陈述（Problem Statement）

问题陈述是一句话，逼你回答四个问题：

1. *谁*——这个人叫什么名字、什么来头？
2. *要什么*——他具体需要什么体验？
3. *为什么*——他为什么有这个需要？

公式像填空：

  [名字] 是一个 [什么样的人]，需要 [什么东西]，因为 [洞察]。

课程给了几个例子：

- *Arnold* 是一个忙碌的高管，需要轻松雇佣遛狗者，因为他不擅长用智能手机。
- *Anika* 是一个忙碌的实习生，需要协作点咖啡的方式，因为单独点单太浪费时间。
- *Ali* 是一个远程学生，需要不离座下单，因为起来可能丢座位。

注意这句话的微妙之处：它不能太宽（"让用户开心"——没法下手），也不能太窄（"加一个蓝色按钮"——把解决方案提前塞进去了）。它要刚好卡在"知道往哪走，但不知道具体怎么走"的位置。

怎么确保卡准？课程给了 **5 Ws and H** 当检查清单：

| 问题 | 大白话 |
|------|--------|
| Who | 到底是谁在受苦？ |
| What | 他到底要什么？ |
| Where | 他在哪用产品？ |
| When | 什么时候出问题？ |
| Why | 为什么这对他重要？ |
| How | 他现在怎么解决的？ |

把六个问题答完，那句话基本就自然出来了。

** 第二步：猜解决方案（Hypothesis Statement）

问题写清楚了，但别急着画草图。先做一个"有根据的猜测"——如果你做了 X，用户就能得到 Y。

两种写法：

1. *If/then*：如果 [用户做了某事]，那么 [他会得到某结果]。
   例：如果 Arnold 下载了 App 并开启简化模式，那么他就能轻松雇佣遛狗者。

2. *We believe*：我们相信 [某方案] 对 [某用户] 能让他们 [某结果]。
   例：我们相信简化模式能让 Arnold 高效雇佣遛狗者。

区别：If/then 聚焦用户行为；We believe 把团队视角也带进来。选哪个看语境，没有标准答案。

关键心态：这是*猜测*，不是承诺。随着你了解变多，这句子会被推翻、改写。它的作用是给你一个可以测试的靶子，而不是一张不可更改的蓝图。

** 第三步：说清你为什么值得被选择（Value Proposition）

市面上已经有十个同类产品，用户凭什么选你？价值主张就是回答这个问题。

课程给了四步蒸馏法：

1. *乱贴便签*——把产品所有功能好处全列出来，不评判。
2. *按用户关心的归类*——看看这些功能能归到哪些用户在乎的桶里（比如：易用、专业、便宜、可靠）。塞不进桶的，先扔一边。
3. *对号入座*——每个 persona 最大的痛点，对应哪个价值？
   例：Arnold 最痛的是"不会用技术" → 对应价值"易用的 App 设计"。
4. *削掉平庸*——竞争对手也提供的功能，从价值主张里删掉。剩下的才是你真正独特的东西。

课程举了 Gmail 的例子：2004 年邮箱市场已经很挤，Gmail 的独特价值是"对话视图"和"1GB 存储"——当时别人都没有。其他功能（免费、过滤垃圾邮件）只是门槛，不是差异。

** 第四步：懂人性——机器不懂的，你要懂

前面讲的都是"方法"，但设计最终是给人用的。人不是机器，人有一堆毛病：

| 人的毛病 | 设计机会 |
|----------|----------|
| 没耐心 | TL;DR 摘要 |
| 记不住 | 把重要信息放开头和结尾 |
| 注意力短 | 减少步骤、即时反馈 |
| 需要动机 | 进度条、成就徽章 |
| 会犯错 | 撤销按钮、防误触 |
| 有偏见 | 用熟悉隐喻降低学习成本 |

课程讲了一段历史：两次世界大战前，设计师想的是"让人适应机器"。战争期间，没经过训练的士兵必须开飞机，于是反过来——*让机器适应人*。这就是 Human Factor 的起源。

三个可以直接上手的心理学武器：

1. *Von Restorff Effect（隔离效应）*：一群相似的东西里，最不一样的那个会被记住。这就是为什么"立即购买"按钮永远长得和其他按钮不一样。

2. *Serial Position Effect（系列位置效应）*：人们只记得列表的开头和结尾。导航栏最重要的功能，放最左或最右，别塞中间。

3. *Hick's Law（希克定律）*：选项越多，决策越慢。薯片货架有二十种口味，你反而更难选。给用户少而精的选择，比给全品类更有效。

** 心智模型与反馈回路

人脑子里有一张*心智模型*（mental model）——预测东西该怎么用。你预期推门能出去，结果门后是一堵砖墙，模型就碎了，人就会困惑和愤怒。

*反馈回路*（feedback loop）是给用户的"动作结果"。按了开关，灯亮了（正反馈）；按了开关，没反应（负反馈）。每一次操作都需要某种确认，让用户知道"我刚才那一下奏效了"。

课程特别提醒：用心理学是为了*赋能*用户，不是*操纵*用户。区别很微妙，但很重要。

* 核心概念

** Problem Statement（问题陈述）

一句话：把人、需求、原因拧成一句话，让团队对准同一个靶子。

类比：医生下诊断。不是"病人不舒服"，也不是"给病人开阿司匹林"，而是"35岁男性，长期熬夜，需要改善睡眠质量，因为他白天工作效率断崖式下跌"。

为什么重要？没有它，团队每个人都在解不同的问题。有了它，争论从"我觉得"变成"这能帮到 Arnold 吗？"

** Hypothesis Statement（假设陈述）

一句话：对"怎么解决问题"做一个可以验证的猜测。

类比：科学家做实验前的假设。不是"我知道这个药能治病"，而是"如果给病人用药，那么炎症指标会下降"。猜错了就换方向。

为什么重要？它把设计师从"我觉得好"拉到"我们可以测"。设计不再是玄学，是可以被证伪的。

** Value Proposition（价值主张）

一句话：用户为什么要选你，而不是选别人。

类比：相亲时的自我介绍。不是说"我会做饭、会赚钱、会修电脑"，而是"我能给你安全感，因为你最怕的是不稳定"。后者才是价值主张——功能是原料，价值是卖给谁的答案。

为什么重要？功能列表谁都能抄，价值主张是你的护城河。

* 洞见

真正值钱的不是"写一句话"这个动作，而是这个意识：*问题定义错了，后面的设计再漂亮也是白搭*。Module 3 的本质是教你"在动手之前，先确保你在解对的问题"。

* 博导审稿

- *选题眼光*：抓到了设计流程中最容易被忽略的环节——从"知道了很多"到"知道该做什么"。很多新手跳过 Define 直接画草图，结果方向错了全白干。
- *方法成熟度*：Problem Statement 的公式简单有效，5 Ws and H 是很稳的检查清单。Hypothesis 的 If/then 和 We believe 两种格式给了灵活性。Value Proposition 的四步蒸馏法尤其实用，从发散到聚焦有清晰路径。
- *实验诚意*：案例给得扎实（Arnold、Anika、Ali、Gmail、Dog Walker App），不是空泛说教。
- *写作功力*：心理学部分的二战飞机历史讲得很生动，让人记得住 Human Factor 的来源。但三个心理学效应（Von Restorff、Serial Position、Hick's Law）讲得偏浅，每个只给了一个例子，没有深入讲怎么组合使用。
- *判决*：weak accept。作为入门课，框架完整、案例鲜活。但学完只会"知道有这几个效应"，距离"在设计中本能地调用它们"还有距离，需要自己额外补实战。

* 启发

- *迁移*：Problem Statement 的填空公式可以搬到任何需要对齐团队的场景——写项目立项书、做产品规划、甚至写邮件汇报。具体做法：先写"谁+要什么+为什么"，再发给大家看，让所有人先对齐这一句话，再讨论怎么做。
- *混搭*：Value Proposition 的四步蒸馏法可以和竞品分析混搭。第一步先列自己的功能，第四步去翻竞品的一星评论，看用户骂什么——骂的点就是你的独特价值机会。
- *反转*：课程默认"先有问题陈述，再有假设"，但现实中有时候你看到一个很酷的技术或交互，倒过来找它能解决什么问题（solution looking for a problem）也能做出好产品。该停下"必须先有问题"的执念，有时候"先有一个好玩的假设，再去找对应的问题"也是有效的创新路径——只是风险更高，需要更快验证。
