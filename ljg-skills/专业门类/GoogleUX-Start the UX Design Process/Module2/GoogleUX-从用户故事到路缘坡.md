> 整理日期：2026-04-24

## 锚点：帮闺蜜订一束生日花

想象你在伦敦的闺蜜 Iyla，想给住在加州的朋友 Priya 订一束生日花。她打开一家加州花店的网站，选花、写卡片，一切顺利——**这是快乐路径**。但到填地址时，系统强制要求选"所在州"，Iyla 住在英国，根本没有州可选。她卡死了，只能关掉网站另找别家。

这个场景串起了本模块的全部核心工具：**用户故事**帮我们定义 Iyla 需要什么；**用户旅程图**帮她找到从"想买花"到"花送到"的每一步；**边缘案例**提醒我们 Iyla 这样的国际用户存在；**无障碍设计**则让我们意识到——帮少数人解决的障碍，往往最终帮到所有人。

## 一、用户故事：用一句话把"谁要什么"讲清楚

### 1.1 它是什么

用户故事就是从用户视角写的一句虚构小故事。目的不是文学创作，而是让设计团队停止猜测，开始对齐。

**固定句式：**

> **作为** [某种用户]，**我想要** [某个行动]，**以便** [获得某种收益]。

| 成分 | 大白话 | 例子（CoffeeHouse 的 Anika） |
|------|--------|------------------------------|
| 谁   | 这个人是谁，什么处境 | 作为每周两次给团队带咖啡的营销实习生 |
| 要什么 | 她想做什么 | 我想提前提交并追踪团体订单 |
| 为什么 | 她到底图什么 | 以便我管好订单准确性，规划好取餐时间 |

### 1.2 为什么用户故事管用

- **排优先级**：需求太多时，故事帮你判断哪些痛点最痛。
- **对齐团队**：一句人人能懂的话，比十页需求文档更能让工程师、设计师、产品经理坐到一起。
- **说服老板**：你给老板看的不是"我们要加个追踪功能"，而是"Anika 每周两次因为订单出错被团队白眼，我们能不能帮她？"

### 1.3 写好故事的 checklist

1. **谁**要具体，带处境。不要"作为一个用户"，要"作为经常出差的狗主人"。
2. **要什么**是行动，不是解决方案。写"我想信任遛狗师"，而不是"我想看五星评分"。
3. **为什么**要触及真正的收益，不是功能本身。"以便我能安心度假"比"以便我能查看状态"更有力量。
4. **多样化**。Anika 是带咖啡的实习生，Ali 是在咖啡店待一下午的远程学生。同一款产品，不同人的故事完全不同。

## 二、快乐路径与边缘案例：别只设计给"正常用户"

### 2.1 快乐路径（Happy Path）

就是一切按预期发展。Iyla 选花、填地址、付款、收到确认短信——流程走完，皆大欢喜。

### 2.2 边缘案例（Edge Case）

事情脱轨了，而且**用户自己修不了**。Iyla 被"州"字段卡住，她不是不会操作，是产品设计根本没考虑她这种情况。这就是边缘案例。

Google Messages 设计师 Ayan 分享过一个真实例子：他们主推 RCS（新型短信），但用户手机信号差时会回退到旧版 SMS。设计团队最初把两者当完全独立的体验来设计，后来发现用户截图里经常出现 RCS 蓝泡泡和 SMS 浅蓝泡泡混在一起。**他们提前识别了这个边缘案例**，让两种状态在视觉上依然和谐——结果这个"小概率情况"在实际中频繁发生。

### 2.3 三个提前抓边缘案例的方法

| 方法 | 具体怎么做 |
|------|-----------|
| ** persona 要够杂** | 你的用户故事如果覆盖了国际用户、残障用户、老年人、新手，边缘案例会自己冒出来 |
| **上线前把自己当用户走一遍** | 不是检查功能有没有，是检查每一步有没有"死胡同" |
| **画线框图** | 视觉化流程后，卡点会变得很明显 |

## 三、用户旅程图：把"她经历了什么"画出来

### 3.1 它是什么

如果说用户故事是一句剧情简介，旅程图就是**分镜脚本**。它记录用户从开始到目标完成的每一步，以及每一步她在想什么、感受什么。

### 3.2 以 Jamal 为例

Jamal 来自密西西比小镇，坐轮椅，第一次去纽约，想上帝国大厦观景台。

**他的任务清单：**
1. 确定地铁路线和方向
2. 找到有轮椅通道的最近地铁站
3. 买票
4. 找到正确站台，确认方向是对的
5. 上地铁
6. 找到正确的出口

**每个任务再拆小动作：**
- 看地铁线路图 → 找无障碍标识 → 确认电梯位置
- 买票 → 看屏幕选项 → 支付
- 等车 → 确认车头方向 → 上车

**他的情绪曲线：**
- 第一次看线路图：困惑、紧张
- 发现电梯坏了要找坡道：焦虑、感到被排除在外
- 终于到观景台：释然、成就感

### 3.3 旅程图的三个价值

1. **杀死设计师偏见**：你不坐轮椅，你可能根本不会想到"电梯坏了"这个分支。画出来，你就被迫看见了。
2. **暴露隐藏痛点**： biracial（混血）的 Kindred 填 census 表时发现没有"混血"选项。这个痛点只有当你真的走一遍她的流程才会浮现。
3. **指向改哪里**：Jamal 的旅程告诉你，地铁图上加一个"无障碍标识"能解决大问题。这个改进不是拍脑袋想出来的，是从旅程里"长"出来的。

### 3.4 画旅程图的四步走

```
Step 1: 列出达成目标的所有主要任务
    |
Step 2: 把每个任务拆成更细的动作
    |
Step 3: 标注用户每步的情绪（可以用表情符号）
    |
Step 4: 在情绪低谷处找改进机会
```

## 四、无障碍设计：为少数人设计，惠及所有人

### 4.1 残疾不是"他们"的事

 accessibility（无障碍）不是只给"残障群体"的。一个人的受限状态可能是：

| 类型 | 例子 |
|------|------|
| **永久性** | 只有一只手 |
| **临时性** | 手腕扭伤 |
| **情境性** | 抱着哭闹的婴儿，只能用一只手操作手机 |

三个人都需要单手操作手机。**设计给 Betty（永久性），顺便帮到了 Angela（临时性）和 Juan（情境性）。**

### 4.2 路缘坡效应（The Curb-Cut Effect）

1945 年，美国密歇根州第一次安装路缘坡（ sidewalk 和马路之间的斜坡），是为了让轮椅使用者通行。但今天，推婴儿车的父母、拉行李箱的旅客、骑自行车的人、老人——**所有人都在用**。

同理：
- **字幕**是给听障人士的，但你在嘈杂酒吧看新闻时也靠它。
- **大按钮、高对比度**是给视力受限者的，但阳光下看手机的人也需要。
- **延长表单填写时间**是给神经多样性用户的，但找信用卡的普通人也受益。

### 4.3 四个感官维度的设计提示

这些不是标准答案，是让你开始思考的起点：

**触摸 / 操作**
- 按钮大小要考虑不同手型
- 支持双击防误触
- 兼容键盘操作（不只是鼠标）
- 允许自定义按钮位置

**视觉**
- 字体别太小，对比度别太低
- 图片和图标加文字说明（alt text），让屏幕阅读器能读出来
- 别只用颜色传达信息（比如红色警告要配文字说明）
- 文字显示方式可自定义（字体、大小、间距）

**听觉**
- 别只靠声音提醒，加振动（触觉反馈）和视觉提示
- 所有视频加字幕
- 提供文字聊天替代语音通话

**言语**
- 提供文字版的内容介绍和说明
- 电话沟通时支持实时文字（Real-Time Text）
- 用表情和图片降低表达门槛

**神经多样性（如阅读障碍、ADHD）**
- 导航保持一致，别让用户重新学习
- 用熟悉的图标，图文搭配
- 流程尽量简单直接

### 4.4 一个立即能做的动作

关掉鼠标，只用键盘操作一次你正在做的产品。Tab 键切换、Enter 键确认。如果你卡住了，那就是一个真实的问题。

## 五、本模块能带走的三样东西

1. **写用户故事时，先问"谁"，再问"要什么"，最后逼自己说出"为什么"。** 三个都清楚了，需求才不会跑偏。

2. **设计流程时，先画快乐路径，再专门留时间找边缘案例。** 方法是：让你的 persona 足够多样化，然后把自己代入每个角色走一遍。

3. **每次做设计决策时，想一下路缘坡。** 你做的这个改进，是不是先帮到了某个少数需求，然后顺便让所有人更爽？如果是，你大概率走对了。
